# 阶段3发布页勘探方案

## 目标

本轮聚焦阶段3最关键、最不稳定的发布链路，只采集后续编码必须依赖的信息：

- 如何稳定进入发布页
- 如何稳定上传 3 张图片
- 如何稳定填写标题与正文
- 如何稳定添加商品
- 如何判断“真的发布成功”
- 失败时应该保留哪些证据

本轮重点不是一次性做完整 50 篇发布，而是为 **单商品单篇发布最小闭环** 建立可编码的页面契约。

本轮产出直接服务这些文件：

- `src/xhs_poster/merchant.py`
- `src/xhs_poster/phase3.py`
- 后续 `rotation-state.json` / `publish-log.json` 相关逻辑

## 为什么现在做这轮勘探

根据当前实施顺序，阶段3应当早于阶段2做探测和最小闭环，因为：

- 发布页是系统里最脆弱的环节
- 它决定了 `contents.json` 未来的消费格式
- 如果先深做阶段2，后续阶段3对标题/正文/商品关联格式有额外要求，会导致返工

## 勘探范围

### 页面 A：发布入口

目标确认项：

- 发布页真实入口是什么：
  - 商品列表页的 `去发布`
  - 商品详情页的某个入口
  - 直接打开 `app-note/publish?...` 形式的 URL
- 发布页是否会自动关联商品，还是必须手动添加
- 打开后是否总会先进入“智能创作”页，再点 `手动创作`
- 发布页 URL 规则是否稳定，是否包含商品 ID 或 source 参数

### 页面 B：发布表单

目标确认项：

- `手动创作` 是否稳定存在，什么时候需要点击
- `上传图文` tab 的真实可点击元素和等待点
- 图片上传控件是否可直接 `set_input_files`
- 标题输入框的稳定选择器
- 正文编辑区是否是 `textarea`、`contenteditable` 或编辑器组件
- 添加话题的最小稳定路径
- 正文中输入 `#` 后是否会弹出标签候选图层
- 标签候选图层的 DOM 结构、定位方式、默认选中项和键盘/鼠标选择方式
- 输入关键词（如 `发夹`）后，候选列表是前端过滤还是触发后台请求
- 选择标签后，正文中的最终落地形式是什么（纯文本、特殊节点、富文本标记）
- 添加商品的真实流程：
  - 自动关联
  - 搜索商品 ID
  - 勾选商品
  - 保存

### 页面 C：发布提交与结果

目标确认项：

- 点击 `发布` 前有哪些必填校验
- 发布按钮的禁用/可点击状态如何判断
- 发布成功后最可靠的信号是什么：
  - URL 跳转
  - 成功 toast
  - 返回笔记列表
  - 页面元素变化
- 发布失败时有哪些可结构化提取的信息：
  - 错误提示文案
  - 当前 URL
  - 页面标题
  - 截图

## 勘探步骤

## 第1步：确认发布入口

我会优先验证：

1. 从商品列表页点击 `去发布` 是否稳定
2. 是否会新开页面 / 新 tab
3. 打开后 URL 是否可复用
4. 是否自动带入当前商品

产出：

- `open_publish_page(product_id)` 的建议路径
- 是否需要优先走按钮点击，还是可直接拼装 URL

## 第2步：确认表单真实结构

我会在发布页采集以下证据：

- 当前 URL、标题
- `手动创作` 是否存在
- `上传图文` 的稳定选择器
- 图片上传 input 的真实定位方式
- 标题输入框定位方式
- 正文编辑器定位方式
- `#` 触发的话题候选图层是否稳定出现
- 输入标签关键词时是否有网络请求、接口节流或异步加载
- 选择候选标签后，DOM 中插入的节点形态
- 添加商品弹层的搜索框、列表项和保存按钮

产出：

- `PublishPage` 各方法的候选选择器和回退方案
- 哪些操作适合 `wait_for_selector`
- 哪些操作必须 `wait_for_url` 或等待弹层出现
- `add_topic(topic)` 是否应当实现为“输入 # -> 输入关键词 -> 等候候选列表 -> 选择第一个匹配项”

## 第3步：验证单篇发布最小闭环

这一步只在你明确允许的前提下进行。

最小闭环定义：

1. 打开发布页
2. 上传 3 张本地图片
3. 填写 1 条测试标题和正文
4. 关联 1 个商品
5. 点击 `发布`
6. 验证是否成功

补充验证点：

1. 正文中的标签不是只停留在普通文本，而是真正完成候选选择动作
2. 若不点选候选项，仅输入 `#发夹`，是否会导致发布后标签不生效

如果你暂时不希望真的发出测试笔记，这一步也可以只做到“点击发布前”，先把页面契约摸清楚。

产出：

- `verify_success()` 的实现依据
- 发布失败时 `screenshot_on_failure()` 需要保存哪些证据
- 哪些字段/按钮最容易抖动

## 第4步：记录异常分支

顺手确认这些情况是否需要编码处理：

- 发布页打开后跳到别的中间页
- `手动创作` 按钮不存在或有 A/B 变体
- 图片上传后长时间处理中
- 正文区域不可编辑
- `#` 候选图层不出现、出现延迟、被遮挡，或输入后没有返回候选标签
- 关键词已输入但候选列表为空，是否允许退化为纯文本发布
- 添加商品弹层加载慢
- 点击发布后无反馈
- 发布失败但 URL 未变化

产出：

- `phase3.py` 的错误模型
- `PublishPage` 的等待点与重试策略

## 你需要怎么配合

开始阶段3勘探时，你只需要按这个节奏配合：

1. 如果 merchant profile 过期，我提示你登录时你再登录
2. 如果出现验证码、短信验证、风险弹窗，你手动处理
3. 如果我需要确认是否允许真正发布一篇测试笔记，我会先征求你确认
4. 其余时间由我做只读观察或最低必要扰动勘探

## 这轮勘探结束时应交付的结果

至少包含：

- 阶段3的真实页面流转路径
- 发布页稳定选择器方案
- `PublishPage` 建议接口与方法边界
- 标签候选图层与 `add_topic(topic)` 的真实交互契约
- 发布成功校验信号
- 失败证据保留方案
- 单篇发布最小闭环的验收标准

## 验收标准

满足下面条件，就说明阶段3页面勘探完成，可以进入最小实现：

- 能稳定进入发布页
- 能稳定定位上传图片、标题、正文、商品关联控件
- 能明确 `#话题` 候选图层的触发、过滤、选择与回退策略
- 已明确点击发布前的必要条件
- 已明确发布成功与失败的识别信号
- 已能把页面行为映射为 `PublishPage` 与 `phase3.py` 的实现接口
