# 实施顺序与里程碑

## 结论

当前推荐顺序：

1. 先完整实现 **Phase 1**
2. 立即转入 **Phase 3 探测 + 最小实现**
3. 再实现 **Phase 2**
4. 最后统一做 **工业化优化**

这个顺序优先降低系统级风险，而不是追求三个阶段表面上都“有代码”。

## 为什么这样排

### 先做完整 Phase 1

原因：

- `today-pool.json` 和 `images/` 是后续 Phase 2、Phase 3 的共同输入
- 商品字段结构、图片目录结构、重跑策略会直接影响后面两个阶段
- Phase 1 已经完成关键页面勘探，当前离“完整可用”最近，收尾成本最低

完成标准：

- 能稳定复用 merchant profile
- 能稳定获取前 10 个商品的 `id`、`name`
- 能为每个商品下载前 3 张主图原图
- 输出稳定的 `today-pool.json`
- 明确 `SESSION_EXPIRED`、空列表、图片不足、详情页异常等错误模型
- 明确重跑覆盖策略

### 为什么 Phase 3 要早于 Phase 2

原因：

- Phase 3 是整个系统里最脆弱的环节，UI 变化、输入框、上传、关联商品、发布结果校验都容易翻车
- Phase 2 的最终输出结构，必须服务 Phase 3 的真实输入要求
- 如果先把 Phase 2 做深，再发现 Phase 3 对标题/正文/话题/商品关联字段有特殊要求，会产生返工

结论：

- Phase 3 是当前最值得优先排雷的高风险模块
- 应先用最小成本证明“能发出去”，再决定 Phase 2 输出 schema 的最终形态

### 为什么 Phase 2 放在 Phase 3 后

原因：

- Phase 2 技术风险相对低，替代方案多
- 热帖抓取和 LLM 生成即使后续微调，也不会像发布链路那样牵一发动全身
- 等 Phase 3 最小闭环跑通后，Phase 2 可以更精准地围绕 `contents.json` 的最终消费方式来设计

## 分阶段里程碑

## 里程碑 A：Phase 1 完整可用

目标：

- `src/xhs_poster/browser.py`
- `src/xhs_poster/merchant.py`
- `src/xhs_poster/phase1.py`

需要达到：

- 可稳定运行
- 输出结构固定
- 错误可解释
- 重跑行为明确

建议补齐项：

- `today-pool.json` 是否允许覆盖旧文件，还是默认更新当天文件
- 图片已存在时是否跳过下载，或提供 `force` 模式
- 登录态失效时统一抛出结构化错误
- 补一个最小 CLI 入口或命令入口，避免只靠脚本调模块

## 里程碑 B：Phase 3 探测

目标：

- 不急着做完整 50 篇发布
- 先把单篇发布链路摸清楚

当前阶段3页面勘探计划见：

- `@.cursor/plans/阶段3发布页勘探_4f2b7c1a.plan.md`

需要确认：

- 发布页真实入口
- 是否总会出现“手动创作”
- 图片上传控件真实可操作方式
- 标题输入框、正文输入框、话题添加、商品关联的稳定选择器
- 发布成功后的可靠校验信号：URL、提示文本、列表页回跳等
- 失败时最小证据保留：截图、URL、页面标题、关键 DOM 摘要

产出：

- `PublishPage` 页面对象接口
- 阶段3的错误模型和等待点
- 单篇发布最小验收标准

## 里程碑 C：Phase 3 最小闭环

目标：

- 至少跑通 **单商品单篇发布**

完成标准：

- 读取 `today-pool.json`
- 读取本地图片路径
- 打开发布页
- 上传 3 张图
- 填写标题/正文
- 关联商品
- 点击发布
- 校验成功
- 失败时保存截图和结构化错误

备注：

- 在这一阶段先不追求 50 篇轮转与完整恢复
- 重点是“真的能发成功”

## 里程碑 D：Phase 2 实现

目标：

- 在 Phase 3 输入约束明确后实现 `consumer.py`、`hot_notes.py`、`content_gen.py`、`phase2.py`

完成标准：

- 能抓热帖
- 能产出分析结果
- 能生成可直接供 Phase 3 使用的 `contents.json`
- `contents.json` 字段尽量贴近发布页真实消费格式

## 里程碑 E：工业化优化

目标：

- 把当前“可跑”升级到“稳定可维护”

范围：

- CLI 统一入口
- stdout/stderr / JSON 输出协议
- 结构化错误码
- 发布状态机
- 重试与恢复
- trace / screenshot / logs
- 文档同步
- 目录和模块职责收敛
